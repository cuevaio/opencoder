# Plan: Electric SQL Connection Lifecycle & Cost Optimization

## Problem
Electric SQL shape subscriptions (HTTP long-poll connections) are being leaked in several places, keeping connections open indefinitely and increasing Electric Cloud costs.

## Issues (ordered by severity)

### 1. eventsCollection LEAK in ChatView (HIGH)
**File:** `src/components/chat/ChatView.tsx:74-77`

`useMemo` creates a new Electric collection per `sessionId`, but never cleans up the old one on session change or component unmount. Each leaked collection maintains an active HTTP long-poll to Electric Cloud.

**Fix:** Add a `useEffect` that calls `eventsCollection.cleanup()` on dependency change/unmount:
```ts
useEffect(() => {
    return () => {
        void eventsCollection.cleanup();
    };
}, [eventsCollection]);
```

React's effect cleanup correctly closes over the previous `eventsCollection` value, so the old collection is cleaned up when `sessionId` changes and the new one remains active.

### 2. useChatLayoutContext fallback creates orphan collections (HIGH)
**File:** `src/routes/_authed/chat.tsx:43-58`

The fallback path calls `createSessionsCollection()` on every render when context is missing. In practice, context is always available because `ChatLayout` provides it to all child routes, making this a latent bug. But if triggered, it creates a new Electric shape subscription per render.

**Fix:** Replace the silent fallback with a thrown error (standard required-context pattern):
```ts
export function useChatLayoutContext(): ChatLayoutContextValue {
    const ctx = useContext(ChatLayoutContext);
    if (!ctx) {
        throw new Error(
            "useChatLayoutContext must be used within ChatLayoutContext.Provider",
        );
    }
    return ctx;
}
```

This eliminates the leak entirely and enforces the architectural constraint that chat components must render inside the chat layout.

### 3. sessionsCollection never cleaned up on unmount (MEDIUM)
**File:** `src/routes/_authed/chat.tsx:86-93`

`sessionsCollection` is created once via `useState` in `ChatLayout` but never cleaned up when `ChatLayout` unmounts (e.g., user navigates away from `/chat/*`). The prefetch manager already has cleanup at lines 114-118 -- this is the same pattern.

**Fix:** Add a `useEffect` cleanup for the sessions collection:
```ts
useEffect(() => {
    return () => {
        void sessionsCollection.cleanup();
    };
}, [sessionsCollection]);
```

### 4. Error handler discards context (LOW)
**File:** `src/lib/collections.ts:28-36`

`onElectricError` silences the error and context with `void`, providing zero observability into Electric connection failures. The proxy endpoints log errors server-side, but client-side failures are invisible.

**Fix:** Add `console.error` with the context:
```ts
function onElectricError(
    error: unknown,
    context: ElectricCollectionContext,
    handler?: (error: unknown) => void,
) {
    console.error("[electric-collection]", context.collectionId, {
        shapeUrl: context.shapeUrl,
        sessionId: context.sessionId,
        error: error instanceof Error ? error.message : error,
    });
    handler?.(error);
}
```

### 5. Use parameterized WHERE clauses in shape proxies (LOW-SECURITY)
**Files:**
- `src/routes/api/shapes/sessions.ts:77` â€” `user_id = '${userId}'`
- `src/routes/api/shapes/session-events.ts:82` â€” `session_id = ${session.id}`

The values are server-controlled (auth-derived userId, DB-returned session.id) so exploitation risk is near-zero. But Electric's AGENTS.md recommends parameterized queries as defense-in-depth.

**Fix:** Use Electric's `$1` param syntax:
```ts
// sessions.ts
shapeUrl.searchParams.set("where", "user_id = $1");
shapeUrl.searchParams.set("params", JSON.stringify([userId]));

// session-events.ts
shapeUrl.searchParams.set("where", "session_id = $1");
shapeUrl.searchParams.set("params", JSON.stringify([session.id]));
```

### 6. session-events proxy DB query per long-poll request (NOT FIXING)
Every Electric long-poll request triggers a DB ownership query. This is a PK lookup (sub-ms on Neon) happening every ~20 seconds per active session view. The security benefit outweighs the negligible cost. Adding a cache introduces session-revocation timing issues. Leave as-is.
