# Plan: Switch to Build Mode After Plan Acceptance (v2 — race condition fix)

## Problem
When a user accepts a plan by answering "yes" to the agent's question while in plan mode, the UI stays in plan mode. The next message the user sends will still be in plan mode instead of automatically switching to build mode.

## Current State
A first attempt exists in `ChatView.tsx`:
- Added `modeOverride` state (lines 69-71)
- `handleAnswer` sets `setModeOverride("build")` after the fetch response (lines 244-249)
- `defaultMode` uses `modeOverride` via nullish coalescing (lines 326-327)

## Why It Still Fails (Race Condition)
`ChatFooter` uses `useState(defaultMode)` — the `defaultMode` prop only sets the **initial** value on mount; the component does not react to prop changes after it is mounted.

The original fix relied on `ChatFooter` unmounting (when `pendingQuestion` is non-null) and remounting (when it becomes null) so it picks up the new `defaultMode`. But there is a race condition:

1. User submits answer → `fetch("/api/agent/answer")` starts (async)
2. Electric SQL delivers a new agent event quickly via WebSocket → `displayItems` updates → `pendingQuestion` becomes null → **`ChatFooter` remounts with `defaultMode = "plan"` (modeOverride not yet set)**
3. HTTP fetch response arrives → `setModeOverride("build")` called → `defaultMode` prop changes to "build"
4. But `ChatFooter` is already mounted with `mode = "plan"` and doesn't react to the prop change

## Fix

### Part 1 — Make `ChatFooter` reactive to `defaultMode` changes (`ChatFooter.tsx`)

Add a `useEffect` matching the existing pattern for `defaultModel`/`defaultVariant` (lines 115-125) that syncs `mode` state when `defaultMode` changes:

```ts
useEffect(() => {
    setMode(defaultMode);
}, [defaultMode]);
```

Insert after line 125 (after the `defaultVariant` effect). With this, regardless of whether `ChatFooter` remounts before or after `modeOverride` is set, it will always reflect `defaultMode`.

### Part 2 — Set `modeOverride` before the fetch (`ChatView.tsx`)

Move `setModeOverride("build")` to *before* the `fetch` call so the override is set optimistically, eliminating the race condition even without Part 1:

```ts
const handleAnswer = useCallback(
    async (tokenId: string, answers: string[][]) => {
        if (effectiveSessionRow?.mode === "plan") {
            setModeOverride("build");  // optimistic — before fetch
        }
        const response = await fetch("/api/agent/answer", {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify({ tokenId, answers, action: "answer" }),
        });
        if (response.ok) {
            completedTokens.add(tokenId);
        }
    },
    [completedTokens, effectiveSessionRow],
);
```
