# Plan: Switch to Build Mode After Plan Acceptance

## Problem
When a user accepts a plan by answering "yes" to the agent's question while in plan mode, the UI stays in plan mode. The next message the user sends will still be in plan mode instead of automatically switching to build mode.

## Root Cause
- `ChatFooter` initializes its local `mode` state from the `defaultMode` prop (line 103 of `ChatFooter.tsx`)
- `defaultMode` in `ChatView` comes from the DB session row (`effectiveSessionRow.mode`), which remains `"plan"` until a new message explicitly changes it
- When `ChatFooter` unmounts (during pending question) and remounts (after answer), it re-initializes from the same unchanged `defaultMode`

## Solution

Add a `modeOverride` state in `ChatView` that switches the effective default mode to `"build"` after a question is answered while the session is in plan mode.

### Changes

**File: `src/components/chat/ChatView.tsx`**

1. Add a `modeOverride` state:
   ```ts
   const [modeOverride, setModeOverride] = useState<"plan" | "build" | null>(null);
   ```

2. In `handleAnswer`, after a successful answer submission, check if the current session mode is `"plan"` and set the override to `"build"`:
   ```ts
   if (response.ok) {
       completedTokens.add(tokenId);
       // After accepting a plan, switch UI to build mode for the next message
       if (effectiveSessionRow?.mode === "plan") {
           setModeOverride("build");
       }
   }
   ```

3. Use the override when computing `defaultMode` (line 319):
   ```ts
   const defaultMode = modeOverride ?? ((effectiveSessionRow.mode as "plan" | "build") || "build");
   ```

This works because `ChatFooter` is conditionally rendered (`{!pendingQuestion && ...}`), so it unmounts during the question and remounts after the answer — picking up the new `defaultMode` value as its initial state.
