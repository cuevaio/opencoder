# Plan: Fix Session Event Seq Collision on Continue

## Problem

When a second message is sent to an existing session (via `POST /api/agent/continue`), the same `agent_sessions` row is reused but a fresh `SessionDbWriter` is created with `this.seq = 0`. The new run then tries to INSERT events with `seq = 1, 2, 3...` ‚Äî colliding with events already written by the first run under the same `(session_id, seq)` unique index. The result is either silently-swallowed errors (events dropped) or explicit DB errors that cause the second run to produce no visible output.

## Root Cause

**File:** `src/trigger/lib/db-writer.ts`, line 17

```ts
private seq = 0;  // always resets to 0 ‚Äî never reads the existing DB eventSeq
```

The `SessionDbWriter` constructor never queries `agent_sessions.event_seq` to find where the previous run left off.

## Fix

### 1. Add an async `init()` method to `SessionDbWriter`

Read the current `eventSeq` from the DB and seed `this.seq` from it.

```ts
// db-writer.ts
async init(): Promise<void> {
  try {
    const [row] = await db
      .select({ eventSeq: sessions.eventSeq })
      .from(sessions)
      .where(eq(sessions.id, this.sessionId))
      .limit(1);
    if (row?.eventSeq) {
      this.seq = row.eventSeq;  // start after the last written seq
    }
  } catch (error) {
    logger.warn("Failed to read eventSeq for session continuation ‚Äî starting from 0", {
      error: error instanceof Error ? error.message : String(error),
    });
  }
}
```

### 2. Call `init()` in `run-session.ts` right after constructing the writer

```ts
// run-session.ts ‚Äî after line 55
const dbWriter = new SessionDbWriter(dbSessionId, payload.userId);
await dbWriter.init();  // ‚Üê seed seq from DB before any writes
await dbWriter.writeStatus("Cloning repository...");
```

`init()` is safe for **new** sessions too: if `eventSeq` is `null` (never written), `seq` stays at 0.

## Files to Modify

| File | Change |
|---|---|
| `src/trigger/lib/db-writer.ts` | Add `async init()` method |
| `src/trigger/run-session.ts` | Call `await dbWriter.init()` after construction |

## Why This Is Sufficient

- `close()` already saves `this.seq` to `eventSeq` after every run, so the value is always current.
- The `partSeqMap` starting empty on the new run is correct ‚Äî part IDs from the prior run's events are already persisted; the new run will assign fresh seqs for any new parts, which will be `> eventSeq`.
- The `UNIQUE (session_id, part_id)` index is unaffected ‚Äî part IDs across runs are distinct UUIDs.
