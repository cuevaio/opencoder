# Fix Chat Page State Reloads on Task Start/End

## Problem

When a Trigger.dev task starts or ends, the chat page visibly "reloads" — content flashes, scroll position jumps, and the UI flickers between states. This happens on:
1. New session creation (navigate to session page)
2. Follow-up message (task restart)
3. Task completion (status → idle)

## Root Causes

### RC1: Race condition in `/api/agent/continue` (HIGH impact)
**File:** `src/routes/api/agent/continue.ts:70-92`

The task is triggered FIRST (line 71), then the session status is set to `"running"` (line 82). Between these two steps, the Trigger.dev task is already inserting events into `session_events` while the client still sees `status: "idle"`. When the status UPDATE finally fires, the UI flashes from "idle" (showing ChatFooter) → "running" (showing "Agent is working...").

### RC2: Extra `agent_sessions` UPDATE for `opencodeSessionId` (MEDIUM impact)
**File:** `src/trigger/run-session.ts:128-131`

The task writes `opencodeSessionId` to `agent_sessions` mid-execution. This produces an extra Electric WAL entry → extra client re-render. The `opencodeSessionId` is not used by the client display — it's only needed at export time.

### RC3: No `staleTime` on chat routes (MEDIUM impact)
**Files:** `src/routes/_authed/chat.tsx`, `chat.$sessionId.tsx`, `chat.index.tsx`

The `_authed` layout has `staleTime: Infinity` to prevent `beforeLoad` re-runs. But the chat routes themselves have no `staleTime`. When Electric delivers rapid data changes, TanStack Router may re-evaluate the route tree, causing unnecessary component re-renders or scroll jumps (compounded by `scrollRestoration: true` in the router config).

### RC4: No loading state during initial Electric sync (LOW-MEDIUM impact)
**File:** `src/components/chat/ChatView.tsx`

When navigating to a session, the Electric subscription hasn't synced yet. `sessionRow` is null and `eventRows` is empty, causing a flash of empty content before data arrives.

### RC5 (bonus): Cancel button is broken — auth check bug
**File:** `src/routes/api/agent/cancel.ts:25`

Line 25 checks `currentMeta.clerkUserId` but the task sets `metadata.set("userId", ...)` (run-session.ts:49). Cancel always returns 403.

## Fix Plan

### Fix 1: Reorder status + trigger in `/api/agent/continue` [P0]
**File:** `src/routes/api/agent/continue.ts`

Set `status: "running"` and `lastPrompt` BEFORE calling `tasks.trigger()`. Then write `triggerRunId` in a separate UPDATE after getting the handle. This way the client sees "running" immediately, before any events arrive.

Add error recovery: if `tasks.trigger()` fails, revert status to the previous value.

**Changes:**
- Move the `db.update(sessions).set({ status: "running", lastPrompt: prompt })` call BEFORE `tasks.trigger()`
- After `tasks.trigger()`, write only `triggerRunId` in a follow-up UPDATE
- Add try/catch to revert status on trigger failure

### Fix 2: Defer `opencodeSessionId` write to export [P0]
**File:** `src/trigger/run-session.ts`

Remove the standalone UPDATE at lines 128-131 that writes `opencodeSessionId`. Instead, pass it through to `exportAndPersistSession()` which already writes in a single atomic UPDATE.

**Changes:**
- `src/trigger/run-session.ts`: Remove lines 128-131, pass `sessionId` (the OpenCode session ID) to the export function (it already receives it as a parameter and writes it at `session-export.ts:96`)
- Verify `exportAndPersistSession` already includes `opencodeSessionId` in its UPDATE — it does (line 96)

### Fix 3: Add `staleTime: Infinity` to chat routes [P1]
**Files:**
- `src/routes/_authed/chat.tsx` — add `staleTime: Infinity` to route config
- `src/routes/_authed/chat.$sessionId.tsx` — add `staleTime: Infinity` to route config
- `src/routes/_authed/chat.index.tsx` — add `staleTime: Infinity` to route config

These routes have no loaders, so `staleTime: Infinity` simply tells TanStack Router to never re-evaluate them due to external data changes.

### Fix 4: Add loading state for Electric sync [P1]
**File:** `src/components/chat/ChatView.tsx`

Add an early return with a loading skeleton when `sessionRow` is null (Electric hasn't synced yet). This eliminates the flash of empty content when navigating to a session page.

### Fix 5: Fix cancel endpoint auth check [P2]
**File:** `src/routes/api/agent/cancel.ts`

Change `currentMeta.clerkUserId` to `currentMeta.userId` on line 25 to match what the task actually sets.
